## Affected Components
- agentkeys-mock-server — full unit test suite (original 12 paths + new rendezvous + auth-request paths, below). **All blockchain-side behavior is out of scope for v0** — the mock backend is the only backend implementation v0 tests.
- agentkeys-cli — integration tests via assert_cmd, including the `agentkeys approve <pair-code>` flow and identity linking
- agentkeys-daemon — unit tests via CLI adapter, plus sandbox-runtime hardening tests (below), plus rendezvous long-poll and auth-request decision-await paths
- agentkeys-core — canonical CBOR serialization test vectors for every `AuthRequestType` variant (`auth_request_vectors.json`); these are the contract the v0 mock backend and the future Heima backend must both pass
- agentkeys-provisioner — orchestrator IPC tests with mock subprocess
- provisioner-scripts/openrouter.ts — weekly CI smoke test against live site
- `@agentkeys/daemon` npm package — postinstall script picks the right prebuilt binary for the host arch; test matrix covers linux-x64, linux-arm64, darwin-x64, darwin-arm64
- `install.sh` (sandbox-side) — integration test that runs the script inside a live `agent-infra/sandbox` container and asserts the daemon ends up supervised

## Key Interactions to Verify
- `agentkeys init` → Google OAuth → master session stored in host OS keychain
- Sandbox-side install via `install.sh` or `npx @agentkeys/daemon` (run by the sandbox or its cloud-LLM operator — not pushed by the host CLI): daemon binary is available and starts with no pair code argument (it generates the pair code itself and displays it).
- Pair flow (E2E, child-initiates, universal across all three deployment targets, v0 against the mock backend):
  1. User launches the daemon in the sandbox (bare `agentkeys-daemon` or `npx @agentkeys/daemon` — no pair code argument).
  2. Daemon generates own keypair → calls `open_auth_request(Pair, {daemon_pubkey, scope})` on mock backend → backend returns `{request_id, otp, pair_code}` → daemon calls `register_rendezvous(daemon_pubkey, pair_code)` → long-polls `poll_rendezvous` → displays pair code in terminal / LLM chat: "Pair code: ABCD-EFGH. Approve on your Master device."
  3. User runs `agentkeys approve ABCD-EFGH` on Mac → CLI calls `fetch_auth_request` with the pair code → displays request details + OTP → user confirms OTP matches what the daemon showed.
  4. CLI sends session-authenticated `approve_auth_request` to mock backend (no client-side signing — backend holds master key and signs internally) → backend signs, marks request consumed, mints child session + wallet.
  5. Daemon's long-poll returns with encrypted child-session payload → daemon decrypts into `memfd_secret` fd → writes at-rest mirror at platform-appropriate path. Pair complete.
  6. Post-pair: `agentkeys link agent-A --alias my-bot` creates a human-readable identity link for future recovery.
- Recover flow (child-initiates): user tells cloud LLM *"recover agent-A"* → fresh daemon calls `open_auth_request(Recover, {human_id: "agent-A", new_daemon_pubkey})` → displays recovery code → user runs `agentkeys approve <code>` on Mac → backend re-encrypts wallet + credentials to new daemon pubkey → daemon receives → same wallet, same credentials, no re-provisioning → prior daemon's pubkey is marked revoked.
- Agent-driven provisioning via MCP: agent (e.g., OpenClaw with browser control) calls `agentkeys.provision(service: "openrouter")` via MCP → daemon's provisioner orchestrator spawns Playwright → creates real OpenRouter account → API key encrypted + stored. Human does NOT run a CLI command for this — the agent initiates it autonomously.
- `agentkeys run -- echo $OPENROUTER_API_KEY` → daemon fetches credential → env var injected → child process sees real API key
- `agentkeys read openrouter` → credential returned to stdout
- Scope denial: agent-A tries to read a service not provisioned for it → DENIED

## Sandbox-runtime hardening tests (Round 13, new)

Daemon startup must assert each kernel feature is present (or downgrade gracefully) before accepting any credential. These run inside the sandbox as part of the daemon process's self-test on boot.

1. **`memfd_secret()` smoke test** — call `syscall(SYS_memfd_secret, 0)`; expect a valid fd. On `ENOSYS`, log a warning and fall back to `mlock2`-only mode. Regression-guards the Round 13 positive reversal (Round 12 assumed this would fail; the probe proved it works on linuxkit 6.10).
2. **`mlock2()` + page-residency check** — verify the pages holding the in-memory session key report `VmLck` > 0 in `/proc/self/status`.
3. **`prctl(PR_SET_DUMPABLE, 0)`** — verify `/proc/self/status` shows `Dumpable: 0` after init.
4. **`prctl(PR_SET_NO_NEW_PRIVS, 1)`** — verify `NoNewPrivs: 1`.
5. **Seccomp filter installed** — verify `Seccomp: 2` (filter mode) and, in a unit test, verify the filter denies `ptrace`, `process_vm_readv`, `kcmp`, `keyctl`, `open("/dev/mem")`, `open("/proc/kcore")`.
6. **Capabilities dropped** — verify `CapEff: 0000000000000000` after init.
7. **Landlock is NOT required** — verify the daemon starts cleanly when `landlock_create_ruleset()` returns `ENOSYS` (probe-confirmed case on Docker Desktop linuxkit). Regression-guards the Round 13 negative reversal.
8. **Session file at `/home/gem/.agentkeys/session`** — created with mode 0600, owner `gem`. Not at `/var/lib/agentkeys/` (root-owned, would need broken sudo revocation) and not at `/workspace` (doesn't exist on this image).
9. **Do NOT touch `/etc/sudoers.d/` or `gem`'s sudo rule** — lint/static test forbidding the string `sudoers.d` in the `agentkeys-daemon`, `agentkeys-cli`, and `install.sh` sources. The probe empirically confirmed that writing a sudo-deny rule for `gem` wedges `gem-server`. This is a hard negative test.
10. **supervisord is PID 1 (asserted by install.sh)** — the sandbox-side install script reads `/proc/1/comm` and aborts with a clear error if it doesn't contain `supervisord`, before attempting to modify `/opt/gem/supervisord.conf`.
11. **Host CLI never uses `/v1/file/write sudo:true`** — lint/static test forbidding the string `/v1/file/write` in the `agentkeys-cli` and `agentkeys-daemon` host-side sources. Install is sandbox-side only; the host CLI's only sandbox contact is the daemon's `/agentkeys/pair*` endpoints.
12. **Revocation latency E2E** — after `agentkeys revoke agent-A`, next credential read from the daemon must DENY within the target latency (mock backend: < 1 s; will become ≤ 1 Heima block in v0.1). This is the LOAD-BEARING defense now that UID isolation is off the table.
13. **Image version guard (by the daemon, at startup)** — on boot, the daemon reads the sandbox version from `/etc/os-release` / `gem-server` self-report and logs a warning if it differs from the versions whose runtime posture we have verified (`1.0.0.152` for v0). Any version bump should trigger a re-run of the runtime probe before trusting the hardening assumptions.
14. **Pair handshake replay resistance** — `POST /agentkeys/pair` with a previously-used nonce must be rejected. Pair without a prior `pair-challenge` must be rejected.

## Edge Cases
- Half-provisioned state: agent calls `agentkeys.provision` via MCP, Playwright creates OpenRouter account but `store_credential` fails before the key is saved
- Session expiry during daemon runtime
- Mock backend unreachable during credential fetch
- Duplicate provision: agent calls `agentkeys.provision(service: "openrouter")` via MCP when OpenRouter is already provisioned for this agent → should return existing credential, not create a second account
- DOM selector changes on OpenRouter signup page — Playwright script must be maintained as the signup UI evolves
- `install.sh` run twice inside the same sandbox (idempotent — detects an existing `[program:agentkeys-daemon]` block, leaves it in place if identical, and makes `supervisorctl reread` a no-op).
- Daemon started twice in the same sandbox — second daemon's `open_auth_request` must generate a new pair code; if the user approves the second, the first daemon's session is invalidated.
- `agentkeys approve <code>` with an unknown pair code — must fail fast with "no pending request for this code" rather than silently waiting.
- `agentkeys approve <code>` but the daemon has already timed out (5-min TTL expired) — must return EXPIRED with a clear message to restart the daemon.
- Recover with an unlinked identity — `--recover agent-A` where "agent-A" has no linked identity in the graph → clear error with guidance: "no agent found for 'agent-A'; use the wallet address directly or link an identity first."
- Sandbox image version drift: daemon starts under a newer `agent-infra/sandbox` release where one of the probe-verified features regressed (e.g., `memfd_secret` gone). Daemon must fall back safely, not refuse to start.
- `/opt/gem/supervisord.conf` format not matching what we expect (e.g., maintainers switched to an include-directory layout). `install.sh` must detect and either use the include dir or abort with an actionable error.
- Sandbox in the wedged state observed during probe2 (API hung due to prior sudo-deny experiment). `agentkeys pair` should detect a hung daemon endpoint and recommend `docker restart`.

## Critical Paths (E2E)
1. Full loop: init → launch daemon (bare, no pair code) → daemon displays pair code → user runs `agentkeys approve <code>` on Mac → daemon receives child session → setup → run → verify env var contains real API key
2. Multi-agent isolation: agent-A gets OpenRouter, agent-B does not, agent-B gets DENIED
3. Revocation: revoke agent-A → next credential read returns DENIED within the target latency (< 1 s against the mock backend)
4. Cloud-LLM-class recovery: pair once, link identity (`agentkeys link agent-A --alias my-bot`), provision OpenRouter, kill the daemon, recover to a fresh daemon via `--recover agent-A` (resolved by human-readable alias), verify same wallet + same credentials without re-provisioning. This is the demo that shows the backend-backed state model working end-to-end.
5. OTP cross-device confirmation: launch daemon (it displays pair code + OTP), run `agentkeys approve <code>` on Mac (it fetches and displays OTP), verify the two OTPs match — the human confirmation channel must actually work in practice, not just in theory.
6. Identity linking round-trip: `agentkeys link agent-A --email bot@example.com` → verify the identity graph stores the link → `open_auth_request(Recover, {human_id: "agent-A"})` resolves to the correct wallet address.

## Rendezvous + auth-request tests (v0, mock backend only)

Scope reminder: v0 implements rendezvous and auth-request as mock-backend endpoints. **No blockchain or TEE testing in v0.** These tests pin down the contract so that when the Heima backend lands in v0.1+, it can be swapped in by passing the same test suite (via the shared `auth_request_vectors.json` fixtures in `agentkeys-core`).

### Rendezvous (mock backend)

1. `register_rendezvous` — daemon registers with a valid `pair_code` → returns a `RegistrationToken` → SQLite `rendezvous_registrations` row exists with TTL 5 min.
2. `poll_rendezvous` — with no matching delivery, long-polls until timeout → returns `Ok(None)` cleanly. Daemon retries without error.
3. `poll_rendezvous` — with a matching delivery → returns `Ok(Some(payload))` with ciphertext intact byte-for-byte.
4. `deliver_rendezvous` — CLI delivers with a valid `pair_code` → daemon's long-poll unblocks within expected latency (< 100 ms local, < 1 s across the public VPS).
5. `deliver_rendezvous` with an unknown `pair_code` → returns `NO_MATCH` (does not leak whether the code existed-but-expired vs never-existed).
6. `deliver_rendezvous` twice for the same `pair_code` → second call returns `ALREADY_DELIVERED` (single-use).
7. TTL expiry — register, wait 6 minutes (fake clock), poll → returns `EXPIRED`; delivery attempts after that also return `EXPIRED`. SQLite row is garbage-collected.
8. Pair code collision avoidance — 100k concurrent `register_rendezvous` calls must produce zero duplicate codes; the derivation must draw from a keyspace large enough that collision probability is negligible at v0 scale. Property test with statistical check.
9. Ciphertext passthrough — verify the backend stores and returns `EncryptedPairPayload` bytes unchanged. Negative test: mutate one byte server-side, confirm daemon's decrypt fails (i.e., the backend has no way to silently tamper).

### Authorization-request primitive (mock backend)

10. `open_auth_request` with `AuthRequestType::Pair` → returns `{request_id, otp, ttl, nonce_hash}` with distinct 256-bit nonce, 60s TTL, OTP derived deterministically from the nonce + canonical request details.
11. OTP determinism test — given a fixed nonce and fixed canonical request bytes, `open_auth_request` produces the same OTP across runs. Prevents implementation drift in the HMAC derivation.
12. Canonical CBOR round-trip — every `AuthRequestType` variant in `auth_request_vectors.json` serializes identically across two independent implementations (`ciborium` vs hand-rolled). Required because the backend's internal signature is over the canonical bytes, so any ambiguity breaks verification on the daemon side.
13. `fetch_auth_request` (Master side) — with a valid master session → returns the full request including decoded details for the user to inspect. With an invalid/expired session → 401.
14. `approve_auth_request` — valid session, user confirms → backend signs internally with the user's master key → request marked `CONSUMED`. Child's `await_auth_decision` returns the backend-signed decision.
15. `approve_auth_request` — request was tampered with server-side (one byte of `request_details` mutated between `open` and `approve`) → daemon-side verification of the signed decision fails because the canonical hash doesn't match what the daemon expects. This is the core integrity test: the daemon verifies the backend signed over the *actual* request, not a substituted one.
16. `approve_auth_request` — request already consumed → `ALREADY_CONSUMED`. Double-approval blocked.
17. `approve_auth_request` — request expired → `EXPIRED`. TTL enforcement.
18. `approve_auth_request` — session belongs to a different user (not the parent of this auth request) → `UNAUTHORIZED`. Cross-account approval attempt blocked.
19. `await_auth_decision` — long-poll blocks until approved OR expired OR consumed. Returns distinct error codes for each terminal state.
20. `await_auth_decision` after consumption — returns `CONSUMED`. The child should then destroy its local copy of the nonce.
21. OTP cross-request replay — open request A with OTP `ABC123`, open request B with different details but an artificially collided OTP `ABC123` (force via test seam) → user approves request A → daemon receives decision for A → attempt to use that decision for request B → daemon-side verification rejects because the canonical hash differs. Proves the OTP is a human channel, not a cryptographic key.
22. Nonce uniqueness — 100k concurrent `open_auth_request` calls → zero duplicate nonces (property test).
23. Recover flow E2E (mock backend) — run the full Pair flow for agent-A, store a real OpenRouter credential, kill the daemon, spin up a fresh daemon with `--recover agent-A`, run through `AuthRequestType::Recover`, verify the new daemon sees the same wallet address and can read the same OpenRouter credential without re-provisioning. Verify the old daemon's pubkey is marked revoked server-side.
24. Recover with wrong session — attacker with a different account session tries to `approve_auth_request` for a Recover request targeting agent-A → `UNAUTHORIZED` because the session doesn't match agent-A's owner.
25. Scope change via auth-request — `AuthRequestType::ScopeChange { agent_id, new_scope }` flows through the same primitive, backend updates scope in SQLite, subsequent credential reads respect the new scope.

## Mock Backend Unit Tests (all 12 paths)
1. POST /session/create — valid OAuth → new account + session + wallet
2. POST /session/create — invalid token → 401
3. POST /session/create — existing account → return existing session
4. POST /session/child — valid parent → scoped child session
5. POST /session/child — invalid/expired parent → 401
6. POST /credential/store — valid session + scope → stored (encrypted)
7. POST /credential/store — duplicate agent+service → update
8. GET /credential/read — valid scope → return decrypted credential
9. GET /credential/read — wrong agent → DENIED + audit event
10. GET /credential/read — service not provisioned → 404
11. POST /session/revoke — valid master revokes child → child reads fail
12. DELETE /credential/teardown — revoke sessions + delete credentials
