## Vision

### 10x Check
The 10x version of AgentKeys is: you type `agentkeys setup my-fleet --services openrouter,brave,anthropic,github,notion` and 5 minutes later, 10 agent sandboxes each have their own scoped credentials, their own EVM wallet addresses, on-chain audit trails, and instant revocation from your phone. No .env files, no manual signups, no service account tokens. The credential layer becomes invisible infrastructure.

For 2x the effort of the current plan, the 10x version adds: multi-service provisioning in parallel, a web dashboard for fleet management, and real Heima TEE integration so the security story is not mocked.

### Core Insight
The magic moment is "fresh sandbox, instant keys." Browser automation that creates real accounts on real services and delivers real API keys to agents without human intervention. Everything else (TEE, blockchain, audit) is supporting infrastructure that makes the magic moment trustworthy and scalable.

## Approach Decision

Three approaches were considered:

- **Approach A: Full Stack** — Build all 13 components as designed in `architecture.md`. Rust CLI + daemon + real Heima pallets + browser automation + kernel hardening. Blocked on Kai meeting (Q1/Q2/Q3). XL effort, months of work.
- **Approach B: Minimal Viable Tool** — CLI + browser automation + mock backend mirroring Heima's API. Ship the provisioning magic without Kai. M effort, weeks of work.
- **Approach C: Heima-First Demo** — TEE credential storage only, no auto-provisioning. User imports existing keys. Proves blockchain story but loses the "instant keys" magic.

**Chosen: Approach B — Minimal Viable Tool with mock Heima backend**

Rationale: Ship the provisioning magic first. Mock the blockchain with a backend service that mirrors Heima's API contract. When Kai builds the real endpoints, swap the mock for the real thing. No external blockers for v0.

The mock backend must be "as real as possible" — same API shapes, same session key semantics, same credential storage patterns. This becomes the spec Kai builds against.

## Accepted Scope (v0)

### Components (6 + mock backend)

Component numbers reference `architecture.md` §2 where applicable. The mock backend (#M) is a temporary v0-only component not in the architecture.md inventory; it will be deleted when Heima integration lands in v0.1.

1. **`agentkeys` CLI (#1)** (Rust) — `init`, `setup`, `list`, `read`, `run`, `usage`, `revoke`, `teardown`
2. **`agentkeys-daemon` (#2)** (Rust) — runs as the `gem` UID inside stock `agent-infra/sandbox` (no UID split — see row 1b). In-process hardening that IS enabled on v0: `memfd_secret()` for the runtime key copy (runtime-verified on linuxkit 6.10), `mlock2(MCL_CURRENT|MCL_FUTURE)`, `prctl(PR_SET_DUMPABLE, 0)`, `prctl(PR_SET_NO_NEW_PRIVS, 1)`, a strict self-installed seccomp-bpf filter denying `ptrace`/`process_vm_readv`/`kcmp`/`keyctl`/dev-mem opens, and capability drop to empty after init. At-rest copy at `/home/gem/.agentkeys/session` (mode 0600). Registered as `[program:agentkeys-daemon]` in `/opt/gem/supervisord.conf` (supervisord is PID 1 — confirmed). Serves MCP + CLI sockets.
3. **MCP adapter (#3)** (Rust) — thin, same process as daemon
4. **CLI adapter (#4)** (Rust) — thin, same process as daemon
5. **Provisioner orchestrator (#7)** (Rust) — exposed as an MCP tool (`agentkeys.provision`) on the daemon. When an agent with browser control (e.g., OpenClaw) calls this tool, the orchestrator spawns a Playwright subprocess to auto-create the service account. The human never runs this directly — it's agent-initiated via MCP.
6. **Browser automation: OpenRouter (#8)** (TypeScript + Playwright) — single service for v0. The Playwright script is called by the provisioner orchestrator (component #7) when an agent requests `agentkeys.provision(service: "openrouter")` via MCP. Creates a real OpenRouter account, obtains the API key, and returns it to the orchestrator for encrypted storage.
7. **Mock backend service (#M, temporary)** (Rust, axum) — mirrors Heima API contract: session management, credential storage with mock shielding key encryption, audit log, scope enforcement, **rendezvous relay** (row 9), **authorization-request primitive** (row 10). SQLite storage. Hosted on a small VPS for 10 users. For v0.1 the entire backend moves to Heima as a set of extrinsics; the rendezvous and auth-request state move into TEE-held state accessed via extrinsics.
8. **`@agentkeys/daemon` npm package** (TypeScript wrapper + bundled Rust binary) — ships `agentkeys-daemon` to every environment a cloud LLM assistant can install into. Follows the esbuild / biome / swc pattern: postinstall script picks the right prebuilt binary for the host arch. Entry point is a bare `agentkeys-daemon` (for new pair) or `agentkeys-daemon --recover agent-A` (for recovery). No pair code argument — the daemon generates the pair code itself and displays it. User types in their LLM chat: *"please run `npx @agentkeys/daemon`"* (new pair) or *"please run `npx @agentkeys/daemon --recover agent-A`"* (recovery).

### Identity model (v0)
- Mock EVM wallet addresses as canonical account IDs (generated deterministically from Google identity hash)
- Google OAuth (or passkey) for user authentication (requires GCP OAuth client setup; limited to 100 test users without Google verification, sufficient for v0)
- **Master key lives in the backend, not on the user's Mac.** After Google/passkey auth, the mock backend generates (or looks up) the user's master keypair and holds the private key server-side. The Mac CLI stores only the **session key** in the host OS keychain (macOS Keychain / libsecret / Credential Manager). The session key proves "I'm the user who authenticated via Google/passkey" and authorizes the backend to sign on the user's behalf. In v0.1 the master key moves into the Heima TEE (same flow, stronger trust model — the operator can't read the key). **Optional local export of the master key to the user's Mac is future work** (see "Deferred to v0.2+").
- Agent session token: in-memory copy held in a `memfd_secret()` fd inside the daemon (runtime-verified working on stock `agent-infra/sandbox`), with an at-rest mirror at `/home/gem/.agentkeys/session` on stock sandbox or `$HOME/.agentkeys/session` on cloud LLM environments (mode 0600). Path choice rationale: `/var/lib/agentkeys/` would require the broken sudo-revocation init-container path, and `${WORKSPACE}` from earlier drafts doesn't exist on this image (`/workspace` missing; the workspace IS `/home/gem`).
- Simple string aliases ("agent-A") mapped to wallet addresses client-side

### Deployment targets (v0)

AgentKeys v0 targets three classes of sandbox, all via the same CLI and the same backend trait:

1. **Local Docker (stock `agent-infra/sandbox`).** User runs it on their Mac. Install is documented as a one-time setup step — the user pastes an install one-liner into the sandbox's own shell (VSCode terminal on port 8200, or `POST /v1/shell/exec`). No host CLI privilege crosses the wire.
2. **User-controlled cloud VM (Fly.io / own Docker build / any self-hosted Linux).** Install via the sandbox's own entrypoint or cloud-init running `curl -fsSL https://get.agentkeys.dev/install.sh | sh`.
3. **Hosted cloud LLM assistant (ChatGPT sandbox + MCP, Claude.ai code execution, Kimi Claw, Manus, Genspark, etc.).** User cannot run shell commands — only chat with the agent. Install happens via a natural-language command to the agent: *"please run `npx @agentkeys/daemon`"* (new pair) or *"please run `npx @agentkeys/daemon --recover agent-A`"* (recovery). The daemon generates the pair code itself and displays it in the chat. The agent uses its own package-install / code-execution tools. The daemon ships as an npm package wrapping prebuilt Rust binaries. Lifecycle is ephemeral per chat session by design.

The pair flow is **identical across all three** because it uses the rendezvous relay built into the `CredentialBackend`, not a direct host→sandbox HTTP connection.

### Auth flow (v0)

Install and pair are **separate responsibilities**. The host CLI never pushes the daemon binary into a sandbox — the sandbox (or its operator agent) is responsible for having `agentkeys-daemon` running, just like it's responsible for having Python or Chrome. The CLI only drives the consent handshake, and that handshake goes through the backend's rendezvous, not a direct network route to the daemon.

**Install** — sandbox-side, per deployment target above. The install step never carries secrets, never elevates host CLI privilege, and is auditable (the script is public). For stock `agent-infra/sandbox`, `install.sh` additionally verifies supervisord is PID 1 and writes `[program:agentkeys-daemon]` to `/opt/gem/supervisord.conf`; it does NOT touch `/etc/sudoers.d/` or alter `gem`'s sudo rule (doing so wedges `gem-server`, see `../../research/aiosandbox/agent-infra-sandbox-runtime-probe.md` §6). For cloud LLM assistants, the daemon runs as a foreground process of whatever REPL/subprocess the agent spawned, no supervisord assumption.

**Pair — child initiates, Master approves (same direction as Chromecast, OAuth device flow, Signal device linking):**

`agentkeys init` is a prerequisite (one-time): Google OAuth in browser → mock backend creates account + mock wallet → returns master session → host OS keychain.

1. **User starts the daemon in the sandbox.** No pair code needed from the Mac — the daemon generates it itself. For the three deployment targets:
   - Local Docker: *"paste into sandbox terminal: `agentkeys-daemon`"*
   - Cloud VM: *"cloud-init runs `agentkeys-daemon`"*
   - Cloud LLM: *"please run `npx @agentkeys/daemon`"*
2. **Daemon initiates the auth request.** On startup, the daemon generates its own keypair and calls `open_auth_request(Pair, {daemon_pubkey, requested_scope})` on the backend. Backend returns `{request_id, otp, pair_code}` (8-char base32, single-use, 5-minute TTL). Daemon also calls `register_rendezvous(daemon_pubkey, pair_code)` and begins long-polling `poll_rendezvous`.
3. **Daemon displays the pair code.** In the sandbox terminal or LLM chat: *"Pair code: ABCD-EFGH. Approve on your Master device."* The code flows FROM the daemon TO the user — one direction, one copy.
4. **User approves on the Mac.** Runs `agentkeys approve ABCD-EFGH`. Mac CLI calls `fetch_auth_request` on the backend using the pair code, displays the request details (type: Pair, requested scope, daemon pubkey fingerprint) and the OTP, and asks the user to confirm the OTP matches what the sandbox showed.
5. **User confirms.** CLI sends a session-authenticated `approve_auth_request` call to the backend — **no client-side signing**. The Mac only holds a session key, not the master private key. The backend holds the master key and signs on the user's behalf.
6. **Backend signs internally and releases the payload.** The backend uses the user's master key (held server-side in v0, inside the TEE in v0.1) to sign the canonical request hash, marks the request consumed, mints a child session + wallet address, and delivers the signed authorization to the waiting daemon via the rendezvous relay. Daemon decrypts into its `memfd_secret` fd and writes the at-rest mirror (`/home/gem/.agentkeys/session` on stock sandbox; `$HOME/.agentkeys/session` on cloud LLM). Pair complete.
7. **Post-pair: link a human-readable identity.** Daemon prompts (or user runs on Mac): `agentkeys link agent-A --email bot@example.com` or `--alias my-openrouter-bot`. This creates an entry in the identity graph (reuses `pallet-omni-account` identity linking). Used for recovery — see below.

**Why child-initiates is correct:** the daemon is the requester (it needs credentials), the Master is the authority (it approves). The OTP flows from the less-trusted side to the more-trusted side, which is how every device-pairing protocol works. The Mac CLI never needs to generate a code upfront — it just responds to incoming approval requests. This also eliminates the `--pair-code` argument from the daemon's CLI, simplifying the install command to a bare `agentkeys-daemon` or `npx @agentkeys/daemon`.

**No host→sandbox HTTP connection required.** The pair handshake goes through the backend's rendezvous relay. No public URL, no port exposure, no firewall punching. Works identically for local Docker, cloud VMs, and hosted cloud LLM sandboxes. The backend sees only ciphertext for the child session payload (encrypted to `daemon_pubkey` before delivery), so the rendezvous relay is not a new trust root.

**Recover (ephemeral daemon case) — child initiates, Master approves:**

Cloud LLM sandboxes are typically ephemeral per chat session. When the session dies, the in-sandbox daemon dies with it and its at-rest session mirror vanishes. **But the wallet, credentials, and scope are in the backend/TEE, not in the daemon** — so the next session just needs to re-attach them to a fresh daemon. No need to re-provision OpenRouter, no need to mint a new wallet, no need to re-approve scope from scratch.

1. User opens a new chat. Types: *"please run `npx @agentkeys/daemon --recover agent-A`"* — using the human-readable alias or linked email set up after the first pair. (Without a linked identity, `--recover 0x9c3e...f4a2` with the raw wallet address also works.)
2. **Fresh daemon initiates recovery.** Generates a new keypair, calls `open_auth_request(Recover, {human_readable_id: "agent-A", new_daemon_pubkey})`. Backend looks up "agent-A" via the identity graph, returns `{request_id, otp, pair_code}`. Daemon registers at rendezvous and displays: *"Recovery code: WXYZ-1234. Approve on your Mac."*
3. **User approves on the Mac.** Runs `agentkeys approve WXYZ-1234`. Mac CLI shows: *"Recover agent-A to a new daemon. OTP: xxxxxx. Confirm? [y/N]"*. User confirms.
4. Backend (or TEE in v0.1) re-encrypts the prior agent's wallet + credential ciphertexts to the **new** daemon's pubkey and delivers via rendezvous. Fresh daemon decrypts and resumes agent-A's state — same wallet address, same scope, same provisioned credentials.
5. The prior daemon's public key is revoked as part of the recover operation, so if it ever somehow comes back (stale replica, retry), it cannot read anything.

Recover is the same primitive as Pair — it just carries a different `AuthRequestType` and the backend performs a re-encryption step instead of minting a new wallet. The human-readable identity link is what makes the UX natural: the user doesn't need to remember a wallet address, just the alias they set up after first pair.

**Identity linking ties into Heima's identity graph.** `pallet-omni-account` already supports linking multiple identities to one account (email, phone, social handles, ENS names). AgentKeys reuses this: `agentkeys link` writes a new identity-link entry for the agent's wallet address. The backend's `open_auth_request(Recover, {human_readable_id})` resolves the link via the same graph. No new infrastructure — just a new use of an existing Heima capability.

**Scope change, high-value release, key rotation** — all route through the same primitive with `AuthRequestType::ScopeChange`, `HighValueRelease`, `KeyRotate`. One mechanism, many uses. Full type list in [`../credential-backend-interface.md`](../credential-backend-interface.md).

- `agentkeys store agent-A openrouter <key>` → manually store a credential scoped to agent-A (the 1Password-like flow, primary CLI demo for humans)
- Agent-driven provisioning (separate demo, via MCP not CLI): an agent with browser control (e.g., OpenClaw) calls `agentkeys.provision(service: "openrouter")` via MCP → daemon's provisioner orchestrator spawns Playwright → browser automation creates real OpenRouter account → API key encrypted to mock shielding key → stored in mock backend. The agent does this autonomously — the human doesn't run a CLI command for provisioning.
- `agentkeys run -- cmd` → daemon fetches credential from mock backend → sets `OPENROUTER_API_KEY=<value>` in child env → execve child
- `agentkeys read openrouter` → daemon fetches credential → prints to stdout (CLI path)
- MCP path also available: daemon exposes MCP tools (`agentkeys.get_credential`) on stdio socket for agent frameworks (Claude Code, Cursor, etc.)

### Mock backend API contract (spec for Kai)
The mock backend exposes these endpoints (will become Heima extrinsics):
- `POST /session/create` — Google OAuth → account + session + mock wallet address
- `POST /session/child` — mint scoped child session for an agent
- `POST /credential/store` — store encrypted credential blob (agent_id, service, ciphertext)
- `GET /credential/read` — fetch credential (enforces scope: agent can only read its own)
- `GET /audit/query` — query audit log by owner/agent/service
- `POST /session/revoke` — kill a session immediately
- `DELETE /credential/teardown` — delete credentials + revoke sessions for an agent

Each endpoint logs to the audit table. This API contract becomes the Heima integration spec.

**Mock backend failure modes:**
- Backend unreachable → daemon retries 3x with backoff, then returns clear error to CLI/MCP caller
- Session expired → daemon prompts re-auth via CLI, MCP returns auth-required error
- Credential not found → returns named error with (agent_id, service) context
- Scope denied → returns DENIED with audit log entry
- Store fails (disk/DB error) → returns 500 with context, orchestrator retries once

## Deferred to v0.1 (Heima integration — blockchain future TODOs)

**Explicit reminder: v0 does not touch the blockchain at all.** Everything below lives in the mock backend for v0. The `CredentialBackend` trait is the contract; v0.1's job is to implement the same contract against Heima without any CLI, daemon, or provisioner changes.

- Real Heima TEE credential storage (replace mock backend)
- Real EVM wallet generation inside TEE
- On-chain audit log (replace mock DB)
- `pallet-secrets-vault` (Kai builds this)
- Scoped child-session key minting in TEE (Kai Q1)
- TEE-side policy enforcement on credential reads (Kai Q3)
- **Rendezvous relay as a Heima extrinsic + TEE state** (added 2026-04-09) — the v0 mock-backend implementation of `register_rendezvous` / `poll_rendezvous` / `deliver_rendezvous` moves to a pallet-backed implementation. Pair codes live in TEE state with a block-counted TTL. The relay ciphertext is already E2E-encrypted to the daemon pubkey, so the pallet only sees opaque bytes. Needs Kai sign-off on "can Heima hold short-lived keyed state with a 5-minute TTL?" — probably yes, but should be confirmed before we lock the v0 trait.
- **Authorization-request primitive as a Heima extrinsic + TEE state** (added 2026-04-09) — v0 mock-backend implementation of `open_auth_request` / `approve_auth_request` / `await_auth_decision` moves to a pallet. The TEE holds the user's master key (migrated from the mock backend's server-side storage), generates the nonce, derives the OTP via HMAC, stores the request with a TTL, signs internally after session-authenticated approval from the user, enforces single-use semantics, and releases the signed decision to the child. The canonical CBOR serialization spec is shared between backends — test vectors in `agentkeys-core/tests/auth_request_vectors.json` must pass against both mock and Heima. Needs Kai sign-off on "can a TEE worker hold a user's master key and sign over a 256-bit nonce with bounded TTL?" — relates to Q1 (scoped sessions).
- **Recover variant's re-encryption step** (added 2026-04-09) — v0.1 requires the TEE to re-encrypt an agent's existing wallet + credential ciphertexts to a new daemon pubkey after verifying a `Recover` auth request. The mock backend fakes this with plaintext handling server-side; Heima has to do it inside the TEE so the shielding key never leaves. This is the load-bearing TEE capability for ephemeral cloud LLM support long-term.

## Deferred to v0.2+
- Kernel hardening — unavailable layers: Landlock (blocked on kernel rebuild with `CONFIG_SECURITY_LANDLOCK=y`), AppArmor/SELinux passthrough (blocked on LSM presence in the container), dedicated `agentkeys` UID + sudo revocation (blocked on hardened-fork of `agent-infra/sandbox` — stock image wedges if we touch `gem`'s sudo). Cheap in-process hardening (memfd_secret, mlock2, prctl, self-installed seccomp-bpf, cap drops) ships in v0, not here.
- Hardened fork of `agent-infra/sandbox` (tracked separately in `1-step-analysis.md` §8.2.5)
- **Master key local export** — optional feature letting the user export their master private key from the backend/TEE to their local machine (OS keychain). Enables client-side signing for auth-request approvals without trusting the backend operator. When present, `approve_auth_request` sends a client-generated signature instead of asking the backend to sign. Not needed for v0 (mock backend is already trusted) or v0.1 (TEE is trusted); becomes relevant for users who want full self-custody.
- **Phone app as Master device** — native mobile app (iOS/Android) that holds the master key (exported from TEE) or authenticates to the backend via its own session. Replaces the CLI prompt for auth-request confirmation with a push notification. ### MCP auth integration — `agentkeys run` as MCP server launcher (v0 demo)

Many MCP tools require authentication (GitHub MCP needs `GITHUB_TOKEN`, Gmail MCP needs OAuth, Notion MCP needs `NOTION_API_KEY`). Today developers hardcode tokens in their Claude Code / Cursor MCP settings. AgentKeys replaces this with scoped, revocable, audited credential injection — zero changes to the MCP servers themselves.

**The pattern:** wrap any MCP server launch with `agentkeys run`:

```json
// Claude Code settings.json — BEFORE (hardcoded token, no revocation, on disk)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_xxxx..." }
    }
  }
}

// AFTER (AgentKeys manages the credential — scoped, revocable, never on disk)
{
  "mcpServers": {
    "github": {
      "command": "agentkeys",
      "args": ["run", "my-agent", "--", "npx", "@modelcontextprotocol/server-github"],
      "env": {}
    }
  }
}
```

`agentkeys run my-agent` fetches the stored credential from the backend, injects it as the appropriate env var (`GITHUB_TOKEN`, `OPENROUTER_API_KEY`, etc.) into the child process's environment, and execs the MCP server. The MCP server reads from env as usual. No protocol changes. No server-side changes. Works with every MCP server that reads credentials from environment variables.

**Why this is the killer v0 demo:**

1. **Store once, use everywhere:** `agentkeys store my-agent github ghp_xxxx` + `agentkeys store my-agent notion secret_yyyy` + `agentkeys store my-agent openrouter sk_zzzz`. Configure Claude Code to launch each MCP server via `agentkeys run`. No tokens in any config file. Ever.

2. **One revoke kills all:** `agentkeys revoke my-agent` instantly invalidates credentials across every MCP server that agent was using — GitHub, Notion, OpenRouter, all of them. One command, total credential death.

3. **Audit trail across tools:** `agentkeys usage my-agent` shows which credentials were accessed, when, by which MCP server process. Visibility that `.env` files never provide.

4. **Zero MCP server changes:** This works with every existing MCP server (GitHub, Notion, Slack, Gmail, Brave, filesystem, etc.) because it operates at the env-var layer, not the MCP protocol layer.

**Three levels of MCP integration (roadmap):**

| Level | How | Effort | When |
|---|---|---|---|
| **L1: Env-var injection** | `agentkeys run my-agent -- <mcp-server>`. Wraps any process. | Zero on MCP server side | **v0 — ships now** |
| **L2: AgentKeys as MCP auth provider** | Agent calls `agentkeys.get_credential(service: "github")` when another MCP tool returns auth-required. Agent orchestrates credential fetch + retry. | Agent-side logic (Claude Code / Cursor would need to recognize auth-needed pattern) | v0.1 — needs agent framework support |
| **L3: MCP auth delegation spec** | Standard MCP mechanism where servers declare auth requirements and a designated auth provider resolves them. Like OAuth for MCP. | MCP spec extension | v0.2+ — AgentKeys could pioneer this as an MCP spec proposal |

**L1 is the v0 scope.** It requires only what we already have (`agentkeys run` is in the CLI command table) plus a documentation section showing the Claude Code settings.json pattern.

### Install script spec (`get.agentkeys.dev/install.sh`)

The install script runs on the user's Mac or Linux machine. It:
1. Detects OS + arch (macOS-arm64, macOS-x86_64, Linux-arm64, Linux-x86_64)
2. Downloads prebuilt binary from GitHub Releases (not `cargo install` — too slow for first impression)
3. Places binary at `~/.agentkeys/bin/agentkeys` and adds to PATH via shell profile
4. Checks prerequisites: Node.js ≥ 18 (for Playwright in `provision` command), Chrome/Chromium (for browser automation). Prints a warning if missing, does NOT fail — `store`/`read`/`run`/`approve` work without them, only `provision` needs the browser.
5. Prints: `✓ agentkeys installed. Run: agentkeys init`

### Auth model note (for docs and README)

**Master setup:** Must be done on your own machine (Mac/Linux). `agentkeys init` uses Google OAuth in a browser. This creates your master session and master key (stored in the backend). Cannot be done via device code from a cloud LLM.

**Child sessions (agents in sandboxes or cloud LLMs):** Authenticate via Gmail (recommended — directly links to a readable identity for recovery) or device code flow. Always scoped. Never master. The cloud LLM is a child that receives scoped credentials, not a manager.

## Sandbox runtime posture (Round 13)

This is the honest v0 security claim for stock `agent-infra/sandbox v1.0.0.152` deployments, derived from the runtime probe. It supersedes any earlier wording about "kernel hardening deferred" — the cheap subset ships in v0, the expensive subset cannot ship on stock sandbox at all.

**What IS enabled in v0:** `memfd_secret()` (runtime-verified, returned `fd=3`), `mlock2(MCL_CURRENT|MCL_FUTURE)`, `prctl(PR_SET_DUMPABLE, 0)`, `PR_SET_NO_NEW_PRIVS`, self-installed strict seccomp-bpf filter (baseline is `seccomp=unconfined`, no conflict), capability drop to empty effective set after init, supervisord-managed lifecycle.

**What is NOT enabled, and why:** Landlock — `CONFIG_SECURITY_LANDLOCK=n` in the linuxkit kernel build (`errno 38 ENOSYS`). AppArmor / SELinux — no LSM operational inside the container (`/sys/kernel/security/lsm` missing). UID split and sudo revocation — `gem-server` (the HTTP control plane) runs as `gem` and shells out to `sudo` for its own privileged operations; empirically verified that writing `/etc/sudoers.d/zz-agentkeys-deny` wedges the entire HTTP API via a self-lock deadlock. No narrower allow-list approach is safe without gem-server source access (the public repo's `docker/` directory is empty).

**Defense in depth reduces to revocation latency:** a compromised agent process can read the session file and daemon memory via `sudo strace`. The only remaining defense is on-chain revocation in ≤ 1 Heima block (~6 seconds once detected) plus a hard 4-hour TTL. This is still a meaningful upgrade over 1Password service-account tokens (no TTL, no per-call signing, no instant revocation) but is strictly weaker than the §3.3a Round 6 design. Documented explicitly because users deploying on other sandbox products (or on the eventual hardened fork) will get stronger guarantees.

1. **Q9 (revocation latency) — NEW TOP PRIORITY after Round 13.** How fast can Heima propagate a session revocation? The runtime probe showed that v0 on stock sandbox has no UID isolation, so revocation latency is our ONLY defense against a compromised agent. If Heima revocation is slow (> 30 seconds), the v0 story weakens dramatically and the hardened-fork TODO becomes a v0 blocker instead of a v0.2 nice-to-have.
2. **Q2 (credential storage):** Will Heima support the credential blob storage shape we're mocking?

4. **Q3 (TEE-side enforcement):** Will the scope enforcement we mock (agent can only read its own creds) be enforceable TEE-side, or only daemon-side?

## Abstract Credential Backend Interface

The mock backend is NOT just a Heima mock. It defines an **abstract credential backend interface** that can be implemented by multiple backends:

1. **Mock backend (v0)** — SQLite, single server, development/testing
2. **Heima TEE (v0.1)** — decentralized, TEE-attested, on-chain audit, the target architecture
3. **Centralized compliant server (future)** — for government/enterprise environments that require centralized control, data residency, or regulatory compliance

The Rust client code in the CLI and daemon should depend on a `CredentialBackend` trait, not on Heima-specific types. The trait surface is:
- `create_session(auth_token) → (session, wallet_address)`
- `create_child_session(parent_session, scope) → (child_session, child_wallet)`
- `store_credential(session, agent_id, service, ciphertext) → result`
- `read_credential(session, agent_id, service) → plaintext`
- `query_audit(session, filter) → Vec<AuditEvent>`
- `revoke_session(session, target_session) → result`
- `teardown_agent(session, agent_id) → result`

This trait is the primary design artifact of v0. Document it as a standalone spec (`credential-backend-interface.md`) so it can be handed to Kai for Heima implementation OR to any other backend implementor.
